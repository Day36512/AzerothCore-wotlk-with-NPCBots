1. Experimental strict creature chase pathing
---------------------------------------------

Files changed:
- src/server/apps/worldserver/worldserver.conf.dist
- src/server/game/Movement/MovementGenerators/TargetedMovementGenerator.cpp
- src/server/game/World/WorldConfig.cpp
- src/server/game/World/WorldConfig.h

New config:
- Creature.Chase.StrictPathing = 0

What changed:
- Added a new world config enum entry:
  CONFIG_CREATURE_CHASE_STRICT_PATHING
- Wired the config into WorldConfig with a default value of false.
- Added documentation in worldserver.conf.dist marking the option as
  EXPERIMENTAL.
- Added a helper in TargetedMovementGenerator.cpp:
  ShouldRejectStrictGroundChasePath(...)
- When the config is enabled, normal non-flying NPCs reject chase paths that
  are:
  - PATHFIND_INCOMPLETE
  - PATHFIND_SHORTCUT
  - PATHFIND_NOT_USING_PATH
  - invalid destination Z
- Pets and player-controlled creatures are excluded from this strict rejection,
  preserving their separate existing handling.

Reasoning:
- AzerothCore's old fallback behavior can turn a failed or incomplete ground
  chase into a direct shortcut toward the target.
- That shortcut is what allows ground creatures to appear to walk through
  terrain, up cliffs, or into the air toward players on unreachable terrain.
- Strict mode intentionally prefers "stop/evade/unreachable" over a fake path.

Behavior observed during testing/discussion:
- Dire Maul North Gordok Hyenas stopped/evaded instead of air-walking when
  strict mode rejected incomplete/shortcut paths.
- An experimental allowance for PATHFIND_INCOMPLETE let the cheating return, so
  it was removed.
- Current behavior is safer: no direct cheating path, but long legitimate routes
  may still stop/evade if the pathfinder reports them as incomplete.

Verification run:
- git diff --check passed.
- cmake --build out\build\x64-Debug --target game --config Debug passed.


2. MMap generator config tuning and documentation
-------------------------------------------------

File changed:
- src/tools/mmaps_generator/mmaps-config.yaml

What changed globally:
- Added comments explaining that the global slope setting should stay
  conservative and that per-map/per-tile overrides are preferred for problem
  areas.
- Changed the documented cell-size example from verticesPerMapEdge 2000 to
  3000.
- Increased global navmesh resolution:
  - verticesPerMapEdge: 2000 -> 3000
- Adjusted dependent global dimensions for the smaller cell size:
  - walkableHeight: 6 -> 9
  - walkableRadius: 2 -> 3
- Left global walkableClimb at 6, with comments explaining that at the new
  3000-vertex scale this is closer to the older effective climb of 4.
- Tightened global simplification:
  - maxSimplificationError: 1.8 -> 1.0

Reasoning:
- Higher navmesh resolution can preserve more terrain detail before
  simplification.
- Smaller simplification error keeps generated paths closer to the original
  geometry.
- Scaling height/radius helps keep similar world-space behavior after increasing
  verticesPerMapEdge.
- The comments steer future tuning toward local overrides instead of making the
  entire world too permissive.

Map/tile override changes:
- Blackfathom Deeps, map 48:
  - cellSizeVertical changed to 0.3556 for the new cell scale.

  - Lumber Mill tile 30,29 keeps walkableSlopeAngle 45 to avoid cliff/pathing

    walkableClimb 9, walkableHeight 8.
- Karazhan, map 532:
  - Added tile 35,52 override with maxSimplificationError 1.0,
    walkableHeight 8, walkableClimb 6.
- The Shattered Halls, map 540:
  - Added map-wide walkableSlopeAngle 70.
- The Blood Furnace, map 542:
  - Added map-wide walkableSlopeAngle 50 and walkableClimb 8.
- The Steamvault, map 545:
  - Added map-wide walkableHeight 8, walkableClimb 7,
    walkableRadius 2, walkableSlopeAngle 70, maxSimplificationError 2.1.

  - Added map-wide walkableHeight 8, walkableClimb 7,
    walkableSlopeAngle 60, walkableRadius 2.
- Blade's Edge Arena, map 562:
  - Changed walkableRadius from 0 to 1, with comments noting the rope/pillar
    tradeoff.

Documentation cleanup:
- Rewrote the RecastDemo/debugOutput workflow comments.
- Clarified how .mmap loc relates to tile names and Recast config values.
- Removed broad example blocks that implied every field could safely be tuned

Important caveat found while reviewing:
- Tile overrides in Config.cpp are parsed as {tileX, tileY}, but looked up with

  extra verification because the lookup appears coordinate-swapped.
- Map-level overrides are safer until that is fixed or confirmed.
